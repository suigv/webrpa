# 功能原子化问题分类说明

更新时间：2026-03-08

## 当前状态

基于已经合入 `main` 的修复结果，这份文档里的问题现在应这样理解：

- `重复实现`：已基本收敛，shared JSON store、selector 生命周期与共享 RPC helper 都已完成主干收口
- `边界混乱`：部分解决，`ui_actions` / `sdk_actions` 已做 facade + 内部分层，其中 `sdk_actions` 已落地为 `sdk_actions.py` facade + `sdk_*_support.py` helper 模式，`core/task_control.py` 目前保留为后续 watchpoint 而非当前阻塞项
- `上层重复编排`：部分解决，`plugins/x_mobile_login/script.yaml` 已对最重复的点击/输入 fallback 链做定向压缩，但仍需继续观察长 workflow 的增长趋势

更完整的已落地结果，见：`docs/reference/功能原子化修复结果.md`

## 一句话结论

当前项目的核心问题不是“插件化方向错了”，而是局部模块开始失去原子化边界：

- 有些能力被不同模块重复实现
- 有些模块把多层职责揉在了一起
- 有些底层抽象不够，导致上层 workflow 重复表达同一种业务动作

这三类问题同时存在，所以代码会越来越复杂，而且后续越改越容易互相牵连。

## 问题分类总表

| 分类 | 含义 | 当前表现 | 典型位置 | 风险 | 应对方式 | 当前状态 |
|---|---|---|---|---|---|
| 重复实现 | 同一种原子能力在多个模块里各写一套 | RPC 接入、连接关闭、参数解析重复出现 | `engine/actions/ui_actions.py`、`engine/actions/state_actions.py` | 一处策略变更需要改多处，容易漂移 | 提取共享 helper / adapter | 部分解决 |
| 边界混乱 | 一个模块同时承担多种层级职责 | 一个文件里既有动作入口，又有策略、连接管理、mini framework | `engine/actions/ui_actions.py`、`engine/actions/sdk_actions.py`、`core/task_control.py` | 模块越来越大，测试和重构成本快速升高 | 按职责拆分内部子模块或 service | 部分解决 |
| 上层重复编排 | 底层缺少合适抽象，导致 workflow 重复写同一业务模式 | 大量 fallback、selector 预加载、点击/输入回退逻辑在 YAML 里反复出现 | `plugins/x_mobile_login/script.yaml` | 插件脚本越来越长，业务改动变得脆弱 | 提炼 composite action，收敛重复模式 | 部分解决 |

## 第一类：重复实现

### 定义

同一个“原子能力”，在多个模块中被重复实现，而不是通过统一 helper 或统一 adapter 复用。

### 当前仓库中的例子

| 例子 | 说明 | 位置 |
|---|---|---|
| RPC 连接参数解析重复 | 不同 action 模块各自维护连接入口、连接关闭、错误处理 | `engine/actions/ui_actions.py`、`engine/actions/state_actions.py` |
| 连接生命周期逻辑重复 | connect / close / fail-safe 模式多处重复 | `engine/actions/ui_actions.py`、`engine/actions/state_actions.py` |

### 这类问题的本质

不是“一个功能被多次说明”，而是“一个能力被多次实现”。

说明文本重复不是核心问题；真正的问题是：

- 行为规则可能不一致
- 错误码和异常处理会慢慢分叉
- 后续升级时必须同步改多处

### 推荐处理方式

- 提取共享 RPC helper
- 将连接管理从 action 逻辑中抽离
- 统一错误码和失败返回格式

### 当前进展

- 已完成：selector 生命周期清理补全，减少了 selector/RPC 资源泄露
- 已完成：shared JSON store 的原子写入、同进程锁和跨进程文件锁加固
- 已完成：共享 RPC helper 已统一抽取到 `engine/actions/_rpc_bootstrap.py`

## 第二类：边界混乱

### 定义

一个模块已经不再是单一职责，而是在同一文件/同一类里混入多个层级的能力。

### 当前仓库中的例子

| 模块 | 目前混合的职责 | 说明 |
|---|---|---|
| `engine/actions/ui_actions.py` | RPC 基础动作 + selector/node 子系统 + app/device 帮助函数 + 截图等能力 | 已接近一个小型子框架 |
| `engine/actions/sdk_actions.py` | SDK action 绑定 + 参数构造 + 共享状态读写 + 部分策略逻辑 | 已不是单纯 action 集合 |
| `core/task_control.py` | 提交、调度、重试、事件、恢复、取消、部分业务反馈逻辑 | 正在演化成总控制器 |

### 这类问题的本质

不是重复，而是“一个模块装了太多东西”。

这会导致：

- 新功能只能继续往旧热点里堆
- 文件越来越长，但又不容易拆
- 一次改动容易误伤多个能力面

### 推荐处理方式

- `ui_actions` 拆成“基础 UI 动作”和“selector/node 子系统”
- `sdk_actions` 拆成内部子模块：保持 facade，按运行时、配置、共享状态、profile、业务辅助分层
- `TaskController` 只保留 orchestration，把业务反馈类逻辑移到 hook/service

### 当前进展

- 已完成：`ui_actions` 已收敛为稳定 facade，并把 selector/node 细节下沉
- 已完成：`sdk_actions` 已收敛为 `sdk_actions.py` facade + `sdk_*_support.py` helper，shared-store、配置读取、profile 与轻业务逻辑已下沉
- 当前结论：`core/task_control.py` 仍需继续控制职责边界，但已不作为本轮 follow-up 的未完成阻塞项

## 第三类：上层重复编排

### 定义

底层原子 action 粒度太低，无法直接表达业务意图，导致 plugin workflow 只能在 YAML 中反复拼接同一套步骤。

### 当前仓库中的例子

| 例子 | 表现 | 位置 |
|---|---|---|
| 点击 fallback 反复出现 | 先 selector 点击，失败再坐标点击 | `plugins/x_mobile_login/script.yaml` |
| 输入 fallback 反复出现 | 先 UI 输入，失败再 shell/设备输入 | `plugins/x_mobile_login/script.yaml` |
| selector 预加载过长 | 一长串 `load_*_selector` 步骤 | `plugins/x_mobile_login/script.yaml` |

### 这类问题的本质

这不是插件写得差，而是下层抽象还不够好。

当同一种业务动作被在 YAML 里展开很多次时，意味着：

- 业务意图没有被沉淀成复合动作
- 插件开始承担本不该由插件承担的低层细节
- workflow 会越来越难读、难维护、难测试

### 推荐处理方式

- 新增 composite action，例如：
  - selector 点击失败自动坐标回退
  - 输入失败自动切 shell/设备回退
  - 批量加载 selector
- 控制超长 workflow 增长，避免继续复制 YAML 模式

### 当前进展

- 当前 `main` 上的 `plugins/x_mobile_login/script.yaml` 仍是长 workflow，但最重复的点击/输入 fallback 链已经复用现有 composite action 做了定向压缩
- 这次不是整插件重写，而是在不改变状态/分支标签语义的前提下收敛重复 YAML 编排
- 如果未来 workflow 再次膨胀到重复 fallback 明显增多，再开启下一轮 compression work 更合适

## 这三个问题的关系

这三类问题不是彼此独立，而是会互相放大：

1. 底层重复实现越来越多
2. 热点模块边界越来越乱
3. 下层抽象不够好，插件只能写更多 fallback
4. 然后更多逻辑又继续堆回热点模块

所以复杂度不是单点增长，而是在形成闭环。

## 你最关心的问题：是不是“一个功能被多次说明了”？

答案是：**部分是，但不完整。**

更准确地说：

- 有一部分是“同一能力被多次实现”
- 有一部分是“一个模块承担了多个能力层级”
- 还有一部分是“底层抽象不够，导致上层重复表达同一种业务动作”

所以不能只理解成“重复说明”或“文档重复”；真正的问题在于代码层的职责分布方式。

## 当前优先级建议

### 第一优先级

1. 持续控制 `TaskController` 的 orchestration 边界，避免新策略回流到热点控制器
2. 若后续控制面继续扩展，优先新增 hook/service，而不是把业务反馈逻辑放回 `core/task_control.py`

### 第二优先级

3. 拆 `engine/actions/ui_actions.py`（已完成主要部分）
4. 持续守住 `engine/actions/sdk_actions.py` 的 facade 边界，新增 support 逻辑优先下沉到 `sdk_*_support.py`（已完成主要部分）

### 第三优先级

5. 继续观察 `plugins/x_mobile_login/script.yaml` 在定向 composite-action 压缩后的增长趋势
6. 后续新增 workflow 时禁止继续复制 fallback YAML 模式（仍然成立）

## 适合直接对外沟通的结论

可以直接复制下面这段：

> 当前项目的复杂度上升，并不只是因为“功能变多了”，而是因为功能原子化边界开始失真。主要有三类问题：第一，同一种底层能力在不同模块里被重复实现；第二，少数热点模块开始同时承担多种职责，形成大而杂的总管模块；第三，底层抽象不足，导致上层插件 workflow 重复写大量 fallback 和编排细节。当前最合理的处理方式不是重写，而是沿着共享 helper、职责拆分、复合动作补齐这三条线逐步收敛复杂度。

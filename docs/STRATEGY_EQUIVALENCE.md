# [DRAFT - PENDING VALIDATION] 动作等效性战略宣言

> [!WARNING]
> 本文档目前处于 **待验证阶段 (Draft)**。其提出的理论尚未经过大规模生产环境验证，不应作为核心执行引擎的硬性逻辑基准。所有的实现必须优先遵循“确定性工程原则”。

## 核心定义：什么是动作的“等效性”？

当我们谈论将一个设备上的操作“蒸馏”并回放到另一个差异巨大的设备上时，我们追求的不是物理轨迹的复刻，而是意图与结果的对齐。

### 1. 空间维度的等效：归一化相对域
**准则**：动作的等效不在于“点击了屏幕的 (X,Y)”，而在于“点击了目标元素所在的相对视觉权重中心”。
- **实施**：禁止在插件中硬编码绝对像素。强制使用 `norm_1000` 坐标体系，通过物理分辨率感知（Physical Resolution Awareness）实现实时转换。

### 2. 语义维度的等效：语义锚点与特征空间
**准则**：动作的等效不在于“点击了写着‘发送’的按钮”，而在于“触发了具有‘提交表单’意图的交互控件”。
- **实施**：
    - 优先使用语言无关特征：`resource_id`、节点层级深度、相邻元素拓扑。
    - 文本参数化：利用 LLM 提取业务参数为 `${vars}`。
    - 视觉抽象：利用 VLM 将控件抽象为“意图目标”而非“文案目标”。

### 3. 结果维度的等效（最高准则）：状态流转 (State-Transition)
**准则**：只要能达成预期的状态跃迁，动作就是等效的。
- **实施**：
    - **状态优先观察**：记录操作前后的 UI 状态 ID (State A -> State B)。
    - **动态决策机制**：如果标准 Selector 失效，回放引擎应根据“当前状态”与“目标状态”的差值，动态决定调用 VLM 定位还是结构化脚本补救。

---

## 应用指引

- **蒸馏工具 (Distiller)**：不应仅聚合 `ui.click`，而应将“点击前后的 UI 状态断言”作为核心。
- **插件开发者**：多使用“轴心对齐”或“ID 定位”，信任引擎的自愈能力。
- **AI 代理**：在生成建议时，始终以“状态完成度”作为第一评估指标。
